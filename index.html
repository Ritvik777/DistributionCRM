Technical Walkthrough

TradeFlow Agent — From part photo to pipeline

Technical documentation for a LangGraph multi-agent platform: identify parts from photos, answer from catalog, draft and send outreach, log to Salesforce CRM—with full observability and evals.

A supervisor routes each request to one of three specialists—GTM, Outreach, or CRM—each backed by its own LangGraph branch. The UI can attach component photos; hybrid vision (CLIP + Claude re-rank + text KB) verifies parts against a catalog before GTM or outreach answers. Downstream nodes retrieve context, call external tools, and enforce business rules such as pricing verification and UI-confirmed send before returning a response. Execution is fully traced with Galileo for debugging, monitoring, and quality evaluation.

Step 1: Architecture Overview

The assistant behaves like a small team with one coordinator and three specialists:

  • Supervisor Routing Agent: reads the user request and picks the best branch (keyword fast-path on the current message + LLM 3-way classifier with CRM/outreach overrides).
  • GTM Agent: product catalog, specs, stock, and pricing (pricing gated behind email verification). Supports component photo verification via hybrid CLIP + vision re-rank.
  • Outreach Agent: formal B2B email drafts, Apollo net-new lead research, Salesforce lead pull for batch outreach, UI-confirmed Brevo send with catalog images — then auto-logs a Salesforce Task.
  • CRM Agent: Salesforce reads, aggregates, DML, schema describe — with fast-path Markdown tables for common lead queries (no LLM tool loop).

Runtime layers: app.py → ui/ → services/ (adapters) → agents/ (LangGraph + tools) → vector_db/ + services/salesforce_*. All branches share one AgentState.

Step 5: Node Graph (Current Runtime)

Node (function) Edge (always) Branch (conditional)

The graph is tree-shaped from a single classifier root. Pricing and send gates are conditional branches. Actual email delivery requires UI send_confirmed=true (Send email button) — the graph's send edge only fires after that. Attached component photos route to GTM via keyword/image fast-path.

Step 7: Node-by-Node Execution Framework

GTM branch logic

  1. gtm_retrieve — component image match (if photo attached) OR KB + web tool loop.
  2. pricing_gate — skipped for component ID; otherwise decides email verification.
  3. collect_email — extracts email from conversation when pricing gated.
  4. gtm_generate — grounded answer with match confidence tiers when applicable.

Outreach branch logic

  1. outreach_research — Apollo net-new leads, Salesforce lead pull, KB/web, component match context.
  2. outreach_generate — formal B2B email drafts (or LinkedIn post) per recipient.
  3. send_gate — "send now" vs "review only" (actual send still needs UI confirm).
  4. outreach_send — Brevo HTML send + catalog image inline/attach + Salesforce Task log.

CRM branch logic

  1. crm_research — fast-path tables (latest leads, time window, part enquiry) OR Salesforce tool loop.
  2. crm_generate — formats tool results (passthrough when fast-path already set answer).

Step 9: Component Vision Catalog

Indexing flow

  1. User uploads catalog photo + optional SKU/name in sidebar.
  2. component_store.index_component_image saves file to data/components/.
  3. CLIP embeds image into Rama_components Qdrant collection.
  4. Claude vision captions the part; caption text is also chunked into text KB.

Match flow (chat attach)

  1. UI pre-runs match_component_image_bytes before calling the agent.
  2. CLIP retrieves top candidates from image collection.
  3. Claude vision re-ranks candidates (shape, markings, package).
  4. Text KB corroborates via caption/SKU search; scores fused into match %.
  5. Results shown in chat panel; GTM/outreach nodes use match context in prompts.

Step 10: Galileo SDK (Tracing + Evaluations)

Tracing in app runtime

  • observability/galileo.py — init, callbacks, spans, sessions, console links.
  • ensure_galileo_initialized() connects project + log stream once.
  • get_langchain_config() injects GalileoCallback into LLM invokes.
  • merge_node_config(...) on every node LLM call for nested span metadata.
  • agents/__init__.py::ask() starts top-level trace; confirm_send() traces UI send.
  • ui/ui.py starts one Galileo session per fresh chat thread.

Evaluation logging flow

  • evals/run_galileo_evals.py uses the same ask() runtime path (no eval/prod drift).
  • Sessions mode: each dataset row opens its own Galileo session via logger.start_session(...).
  • Experiment mode: uses galileo.experiments.run_experiment(...) for experiment tracking.
  • Expected tool coverage is inferred from trace steps and validated at run end.
  • Key env vars: GALILEO_API_KEY, GALILEO_PROJECT, GALILEO_LOG_STREAM.

Step 11: Examples (End-to-End Requests)

Example A — Pricing gate

User: "What does this product cost?"

Path: classify → gtm_retrieve → pricing_gate → collect_email

Outcome: asks for verified work email before full pricing details.

Example B — Component photo

User: attaches component photo + "Is this in our catalog?"

Path: UI match → classify(image) → gtm_retrieve → gtm_generate

Outcome: match % + SKU/stock answer; pricing gate skipped for component ID.

Example C — Content draft

User: "Write a LinkedIn post for distribution teams."

Path: classify → outreach_research → outreach_generate → send_gate(review)

Outcome: returns draft only; no email sent.

Example D — Confirmed send

User: drafts email, clicks Send email in UI.

Path: confirm_send() → outreach_send (bypasses graph draft step)

Outcome: Brevo HTML send + catalog image + Salesforce Task logged.

Example E — CRM fast-path

User: "Show latest 10 leads" / "Leads who enquired about Part No. LED-RED-5MM"

Path: classify → crm_research (no tool loop) → crm_generate passthrough

Outcome: Markdown table from repository fast-path.

Example F — Cross-agent outreach

User: "Email Salesforce leads about LED Red 5mm availability."

Path: classify(outreach) → outreach_research(SF leads) → outreach_generate → send_gate(review)

Outcome: one formal email per CRM lead; held for UI confirm.
